1 Introduction

1.1 Document Overview

When using a POS cash register, the API interface must be called upon successful order payment, successful refund, or order deletion to send the order data back to the cash register system.

1.2 Target Audience

For reference and use by technical or business personnel of the merchant platform service provider.

2 Solution Overview

2.1 Business Implementation Flow

API call sequence:

The Create Order API, Refund API, Order Delete API, and other interfaces all involve a signing process. All calls must be completed on the merchant's server side.

3 Data Format

3.1 Submit Data

Uses the standard HTTPS POST protocol. To ensure the receiver receives data correctly, the transmitted data must be signed.

{

"service":"pay.cash.intl",

"client_id":"1001",

"store_id":"1001001",

"order_no":"4370dcbf793645269e260527aea1ba0c",

"amount":1.0,

"client_ip":"127.0.0.1",

"nonce_str":"1bwpzeoh",

"octopus_no":"10011",

"order_type":"CAT",

"trade_status":"SUCCESS",

"trade_sn":"100111", "pay_finish_time":"2023/12/11 00:00:00",

"sign":"194565B69FD86C12F8148A52E37718C4",

"lang":"en_US"

}

Successful response:

"order_no":"4370dcbf793645269e260527aea1ba0c",

"transaction_id":"FTP202012181655430001",

"submit_time":"20201218165543",

"nonce_str":"xcf5xkkv",

}

When message returns a value other than SUCCESS, it indicates an API call error.

4 Digital Signature

To ensure the authenticity and integrity of data during transmission, digital signatures are required. The signature is verified upon receiving the signed data.

Digital signing involves two steps: first, assemble the original string to be signed according to specific rules; then, apply the chosen algorithm and key to compute the signature result.

4.1 Signature Original String

Whether for requests or responses, the original signature string is assembled as follows: 1. Excluding the sign field, all parameters are sorted in ascending order by field name using ASCII values, then concatenated in QueryString format (i.e., key1=value1&key2=value2…). Empty values are not included and do not participate in the signature string. 2. In the original signature string, both field names and field values use their original values without URL encoding.

3. The platform's response or notification messages may include additional parameters due to upgrades. Please account for this when verifying response signatures.

Example:

Calling an API with the following fields:

{

"client_id":"1001",

"store_id":"1001001",

"service":"pay.cash.intl",

"order_no":"Me08772e914454656bf26ec8996e35bb",

"amount":"1",

"body":"a chair",

"notify_url":"2031",

"payment_inst":"CASH",

"client_ip":"127.0.0.1",

"nonce_str":"1409196838",

"sign":"22C9835251DCF58BB70C78FA8D6448BD"

}

The correct signature field ordering is:

amount=1&body=a chair&client_id=1001&client_ip=127.0.0.1&nonce_str=1409196838&notify_url=2031&order_no=Me08772e914454656bf26ec8996e35bb&payment_inst=CASH&service=pay.cash.intl&store_id=1001001

4.2 Signature Algorithm

MD5 Signature MD5 is a digest generation algorithm. By appending the merchant communication key to the original signature string and performing MD5 computation, the resulting digest string becomes the signature result. For ease of comparison, the signature result is uniformly converted to uppercase characters.

Note: When converting the string to a byte stream during signing, the specified encoding character set must be consistent with the charset parameter. MD5 signature formula: sign = MD5 (original string+&key=merchant secret key). toUpperCase

Assuming the merchant secret key is: MjNiYWE4NGY2YTMxNDk2MTljODBkMjZkMjk0NjAxY2M=

Example: Assuming the following JSON input parameters: i: After step a, the URL key-value pair dictionary-sorted string string1 is: amount=1&body=a chair&client_id=1001&client_ip=127.0.0.1&nonce_str=1409196838&notify_url=2031&order_no=Me08772e914454656bf26ec8996e35bb&payment_inst=CASH&service=pay.cash.intl&store_id=1001001 ii: After step b, the sign is:

sign = md5(string1+&key=merchant secret key).toUpperCase

The actual encrypted string is:

sign = md5(amount=1&body=a chair&client_id=1001&client_ip=127.0.0.1&nonce_str=1409196838&notify_url=2031&order_no=Me08772e914454656bf26ec8996e35bb&payment_inst=CASH&service=pay.cash.intl&store_id=1001001&key=MjNiYWE4NGY2YTMxNDk2MTljODBkMjZkMjk0NjAxY2M=).toUpperCase

The final signature obtained is: 22C9835251DCF58BB70C78FA8D6448BD

The sign and lang fields do not participate in the API signature. The same applies to the decryption rules for API response parameters.

5 API Description

5.1 Create Order API

5.1.1 Business Function

Initialize a create order request. Through this request, the data of successful POS cash register payments is saved to the cash register system.

5.1.2 Interaction Mode

Request: Backend synchronous request mode

Response: Backend request interaction mode

5.1.3 Request Parameters

Request URL: https://api.fintechpaymenthub.com/openapi/v1/pay

(Same as the original FTP PaymentHub Create Order API URL; new additions are highlighted in yellow)

Call the API via POST JSON body

Field Variable Required Type Description
API Type service Yes String(40) Cash payment: pay.cash.intl Octopus payment: pay.octopus.intl
Client ID client_id Yes String(32) Merchant client ID
Store ID store_id Yes String(50) Store ID
Merchant Order No order_no Yes String(32) Order number within the merchant system, up to 32 characters, may contain letters. Must be unique within the merchant system.
Octopus Card No octopus_no No String(32) Octopus card number
Data Source order_type No String(32) Lucky Cat: CAT Cash Register: FTP Defaults to Cash Register if not provided
Payment Completion Time pay_finish_time No String(32) Format: yyyy/MM/dd HH:mm:ss
Trade State trade_status No String(32) Success: SUCCESS Failure: FAIL Defaults to success if not provided
Payment Channel No trans_sn No String(32) Displayed as submitted
Amount amount Yes String(15) Total order amount (must be greater than 0), in yuan.
Terminal IP client_ip Yes String(16) IP of the machine where the order was generated
Random String nonce_str Yes String(32) Random string, no longer than 32 characters. A different value must be submitted each time.
Signature sign Yes String(32) See Chapter 4 "Signature Rules"
Language lang No String(32) Language: zh_HK, zh_CN, en_US

5.1.4 Response

Data is returned immediately in JSON format

Field Variable Required Type Description
Merchant Order No order_no Yes String(32) Order number generated by the merchant
FTP Order No transaction_id Yes String(26) Order number generated by FTP
Submit Time submit_time Yes String(50) Format: yyyyMMddhhmmss
Result Message message Yes String(500) Describes the query result. SUCCESS indicates success; otherwise, other error messages are returned.
Random String nonce_str Yes String(32) Random string, no longer than 32 characters
Signature sign Yes String(32) See Chapter 4 "Signature Rules"

5.2 Refund API

5.2.1 Business Function

Send data of successful refunds back to the cash register system. The cash register system stores this data.

5.2.2 Interaction Mode

Backend system call interaction mode

5.2.3 Request Parameters

Request URL: https://api.fintechpaymenthub.com/openapi/v1/refund

(Same as the original FTP PaymentHub Refund API URL; new additions are highlighted in yellow) Call the API via POST JSON body

Field Variable Required Type Description
API Type service Yes String(40) Cash payment: pay.cash.intl Octopus payment: pay.octopus.intl
Client ID client_id Yes String(32) Merchant client ID
Store ID store_id Yes String(50) Store ID
Merchant Order No order_no Yes String(32) Original order number generated by the merchant
FTP Order No transaction_id Yes String(26) Order number generated by FTP for the original order
Merchant Refund No out_refund_no Yes String(32) Merchant refund number, up to 32 characters, may contain letters. Must be unique within the merchant system. Multiple requests with the same refund number are treated as a single order by the platform and only one refund will be processed. If a refund fails, please resubmit using the original refund number to avoid duplicate refunds.
Remarks remarks No String(255) Refund remarks
Refund Amount refund_fee Yes String(15) Total refund amount, in yuan. Partial refunds are supported.
Octopus Card No octopus_no No String(32) Octopus card number
Refund Completion Time refund_finish_time No String(32) Format: yyyy/MM/dd HH:mm:ss
Refund State refund_status No String(32) Success: SUCCESS Failure: FAIL
Payment Channel No trans_sn No String(32) Displayed as submitted
Random String nonce_str Yes String(32) Random string, no longer than 32 characters
Signature sign Yes String(32) See Chapter 4 "Signature Rules"
Language lang No String(32) Language: zh_HK, zh_CN, en_US

Data is returned immediately in JSON format
