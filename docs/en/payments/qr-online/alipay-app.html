1 Introduction

1.1 Document Overview

Alipay APP Payment is a payment method where the merchant APP uses the third-party SDK to interface with the third party and invokes payment through the order parameters generated by the merchant. After the payment is completed, the personal data will jump back to the merchant's APP.

The merchant APP calls the Alipay SDK to invoke the Alipay APP, and the merchant APP will redirect to Alipay. After the Alipay transaction is completed, it will redirect back to the merchant APP, and finally the payment result is displayed in the merchant APP.

5.3 Refund API

5.3.1 Business Function

The merchant initiates a refund for an order that has been successfully paid, and the operation result is returned synchronously in the same session.

1. Refund Method

Currently only original-route refund is supported.

Note: Refund to bank card is not instant. Each bank has a different processing speed, and generally the refund arrives within 1-3 working days after the refund is initiated.

For partial refunds of the same order, the same order number and different out_refund_no must be set. If a refund fails and is resubmitted, the original out_refund_no must be used. The total refund amount cannot exceed the actual amount paid by the user (the voucher amount cannot be refunded).

2. Refund Restrictions

Merchants should pay attention to the refund restrictions during refund operations to avoid initiating refund requests that will not succeed. The main refund restrictions are as follows:

1. On the platform, as long as the cumulative refund amount does not exceed the total payment amount of the transaction order, a transaction order can be refunded multiple times. The refund application number (this parameter exists in the refund API) uniquely determines a refund, rather than the transaction order number determining a refund. The refund application number is generated by the merchant, so the merchant

must ensure the uniqueness of the refund application number. Merchants should pay special attention during the refund process that another refund can only be re-initiated when it is certain that the refund has failed.

2. Currently, most banks support full refunds and partial refunds, but a few banks do not support full refunds or partial refunds, or

do not support refunds. In this case, the merchant can coordinate with the seller to refund to the Alipay account.

Currently, only the no-key refund API is provided. Merchants who need a key refund API, please contact business for explanation.

5.5 Payment Notification API

5.5.1 Business Function

The payment notification URL is the notify_url parameter submitted in Web payment. After payment is completed, the platform will send the relevant payment and user information to the URL. The merchant needs to receive and process this information.

For backend notification interaction, if the platform receives a response from the merchant that is not the pure string "success" or returns after more than 5 seconds, the platform considers the notification unsuccessful. The platform will intermittently re-initiate the notification through a certain strategy (notification interval: 0/15/15/30/180/1800/1800/1800/1800/3600, unit: seconds) to maximize the success rate of the notification, but cannot guarantee the final success of the notification.

Due to the possibility of resending backend notifications, the same notification may be sent to the merchant system multiple times. The merchant system must be able to handle duplicate notifications in a correct manner.

The platform recommended method is, after receiving the notification, to first check the status of the corresponding business data to determine whether the notification has been processed. If not processed, perform other processing; if already processed, directly return the pure string success. Before checking the status and processing the business data, a data lock should be used for concurrency control to avoid data confusion caused by repeated insertion of data in the function.

Note: After the merchant backend receives the parameter notification. It needs to verify out_trade_no and total_fee in the received notification against the order number and order amount of its own business system. And only after the verification is consistent can the order status in the database be updated.

Suggestion: After creating an order and initiating payment, if no payment success notification request is received within 5 minutes, it is recommended to initiate the order query API, querying once every 5 seconds, for a total of 12 queries.

The backend notification will be returned via the notify_url of the payment request in POST mode.
