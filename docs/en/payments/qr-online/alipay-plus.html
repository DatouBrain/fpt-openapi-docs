1 Introduction

1.1 Document Overview

The A+ Checkout API is a convenient payment method. Merchants call the API interface to query which payment wallets are supported for payment. Then, customers can choose different A+ wallets to make the payment. Depending on the terminal initiated by the merchant, different payment flows will apply. For example, on the WEB terminal, a QR code may be displayed, supporting the user to scan the code and pay using the wallet. On the WAP terminal, the user's wallet APP may be triggered to make a direct payment.

Generally, there are the following scenarios.

(1) On the WEB terminal, a QR code may be displayed to support the user to scan the code and pay using the wallet.

Figure 1 Wallet scan code web payment flow

(2) For some wallets that do not support QR code payment (list as follows): DANA / BPI / Boost / Akulaku PayLater / BillEase, on the web terminal, the wallet's account password login page will be displayed, supporting the user to pay.

Figure 2 Payment flow of entering wallet account password on web terminal

(3) The user opens the cashier link on the WAP terminal, selects the A+ wallet payment, and the wallet APP can be triggered for payment.

Figure 3 Payment flow of WAP terminal redirecting to wallet APP

(4) The user opens the cashier link on the WAP terminal, selects the A+ wallet payment, redirects to the wallet account password login page, and pays after logging in.

Figure 4 Payment flow of entering wallet account password on WAP terminal

(5) The user opens the cashier on the merchant APP or mini-app terminal, selects the A+ wallet payment, and the wallet APP can be triggered for payment.

Figure 5 Payment flow on merchant APP or MINI APP terminal

1.2 Intended Audience

For technical or business personnel of the merchant platform service provider to reference and query.

5.3 Refund API

5.3.1 Business Function

The merchant initiates a refund for an order that has been successfully paid, and the operation result is returned synchronously in the same session.

I. Refund Method

Currently, only the original route refund is supported.

Note: Refunds to bank cards are not real-time. The processing speed varies by bank, and generally, the refund arrives within 1-3 business days after initiation.

For partial refunds of the same order, the same order number and different out_refund_no must be set. If a refund fails and is resubmitted, the original out_refund_no should be used. The total refund amount cannot exceed the actual amount paid by the user (cash coupon amounts cannot be refunded).

II. Refund Restrictions

Merchants should pay attention to refund restrictions during refund operations to avoid initiating refund requests that will not succeed. The main refund restrictions are as follows:

1. On the platform, as long as the cumulative refund amount does not exceed the total payment amount of the transaction order, a transaction order can be refunded multiple times. The refund application number (this parameter exists in the refund API) uniquely identifies a refund, rather than the transaction order number. The refund application number is generated by the merchant, so the merchant

must ensure the uniqueness of the refund application number. Merchants should pay special attention during the refund process that a new refund can only be initiated when it is certain that the refund has failed.

2. Currently, most banks support full refunds and partial refunds, but a few banks do not support full refunds or partial refunds, or

do not support refunds. In this case, the merchant can coordinate with the seller to refund to the Alipay account.

Currently, only the keyless refund API is provided. Merchants who need a keyed refund API, please contact business for explanation.

5.5 Payment Notification API

5.5.1 Business Function

The payment notification URL is the notify_url parameter submitted in Web payment. After the payment is completed, the platform will send the relevant payment and user information to the URL. The merchant needs to receive and process this information.

During backend notification interaction, if the platform receives a response from the merchant that is not the pure string "success" or that responds after 5 seconds, the platform will consider it unsuccessful. The platform will intermittently restart the notification with a certain strategy (notification interval: 0/15/15/30/180/1800/1800/1800/1800/3600 unit: seconds), trying its best to improve the notification success rate, but cannot guarantee the final success of the notification.

Since backend notifications may be resent, the same notification may be sent to the merchant system multiple times. The merchant system must be able to handle duplicate notifications correctly.

The method recommended by the platform is: after receiving a notification, first check the status of the corresponding business data to determine whether the notification has been processed. If not processed, perform other processing; if already processed, directly return the pure string success. Before status checking and processing of business data, a data lock should be used for concurrency control to avoid data confusion caused by duplicate data insertion in the function.

Notification: After the merchant backend receives the parameter notification, it needs to verify the out_trade_no and total_fee in the received notification based on its own business system's order number and order amount. The order status in the database can only be updated after verification consistency.

Suggestion: After creating an order and initiating payment, if no payment success notification request is received within 5 minutes, it is recommended to initiate the order query API, querying once every 5 seconds, for a total of 12 queries.

Backend notifications will be returned via the notify_url that sent the payment request in POST mode.
