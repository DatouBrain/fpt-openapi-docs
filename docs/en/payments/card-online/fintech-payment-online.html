5.3 Refund API

5.3.1 Business Function

The merchant initiates a refund for a successfully paid order, and the operation result is returned synchronously in the same session.

I. Refund Method

Currently, only original path return refunds are supported.

Note: Refunds to bank cards are not instant; each bank has different processing speeds, and generally, refunds arrive within 1-3 business days after initiation. The total refund amount cannot exceed the user's actual payment amount (cash voucher amounts cannot be refunded).

II. Refund Restrictions

Merchants should pay attention to refund restrictions when performing refund operations to avoid initiating refund requests that will not succeed. The main refund restrictions are:

1. The refund application number is generated by the merchant, so the merchant must ensure the uniqueness of the refund application number. During the refund process, merchants should pay special attention: only when a refund failure can be confirmed should another refund be initiated.

2. Currently, most banks support full refunds and partial refunds, but a few banks do not support full or partial refunds, or do not support refunds at all. In such cases, merchants can coordinate with the buyer.

5.5 Payment Notification API

5.5.1 Business Function

After payment is completed, the system will notify relevant data to the notify_url filled in during order creation. The notification URL is the notify_url parameter submitted in the payment API. After payment is completed, the platform will send relevant payment and user information to this URL.

Merchants need to receive and process the information.

For backend notification interaction, if the platform does not receive a pure string "success" response from the merchant or if the response is returned after more than 5 seconds, the platform considers the notification failed. The platform will intermittently re-initiate notifications through a certain strategy (notification frequency is 0/15/15/30/180/1800/1800/1800/1800/3600, unit: seconds) to improve the success rate of notifications as much as possible, but does not guarantee that the notification will ultimately succeed.

Since there are cases of resending backend notifications, the same notification may be sent to the merchant system multiple times. The merchant system must be able to correctly handle duplicate notifications.

The recommended approach is, when receiving a notification for processing, to first check the status of the corresponding business data to determine whether the notification has already been processed. If it has not been processed, proceed with processing; if it has been processed, return a success result directly. Before performing status checks and processing on business data, use data locks for concurrency control to avoid data confusion caused by function re-entry.

Special note: After the merchant backend receives the notification parameters, it must verify the order number order_no and order amount amount in the received notification parameters against the orders and amounts in its own business system. Only after verification of consistency should the database order status be updated. Backend notifications are performed via the notify_url in the request, returning data stream via POST method. The specific information is an XML format string (merchants should pay attention to this when processing).

6 Checkout Payment Mode Integration Guide

📖 Scope

This chapter applies to Fintech Payment Online APIs (pay.master.ftp.online / pay.visa.ftp.online / pay.unionpay.ftp.online) with transaction_type=checkout checkout payment scenarios.

6.1 What is Checkout Mode

Checkout mode means that after the merchant calls the order API on the server side, a platform-hosted checkout page link (pay_url) is returned. The merchant then redirects the user to this page to complete the payment. The user fills in credit card information on the checkout page, and all sensitive data such as card numbers is handled by the platform side. The merchant does not need to collect or store card information in their own system.

6.2 Correct provide_type Value

⚠️ Note

In Checkout mode, provide_type must be set to checkout, not card.

The provide_type field has two values with completely different meanings and validation behaviors:

provide_type Value Meaning Requires Card Info (pan/cvv/expiry_year/expiry_month)
card Merchant has collected card info, passes card number directly Yes, required
checkout Platform checkout hosted, user fills card on redirect page No, not needed

If you incorrectly combine transaction_type=checkout with provide_type=card, the server will forcibly require pan, cvv, expiry_year, expiry_month, even though the checkout flow does not need them. The API will return an error:

{
  "message": "Pan field is required",
  "errors": [
    { "field": "pan",          "code": "MISSING_FIELD" },
    { "field": "expiry_year",  "code": "MISSING_FIELD" },
    { "field": "expiry_month", "code": "MISSING_FIELD" },
    { "field": "cvv",          "code": "MISSING_FIELD" }
  ]
}

6.5 Signature Notes

Checkout mode requests typically contain an items array, which is a nested field requiring the new signature rules (V1.1).

When calculating the signature, the items field must first be serialized into a compact Canonical JSON string (no extra spaces) before participating in ASCII sorting.

Correct example (Python):

import json, hashlib

items = [{"description": "Product name", "quantity": "1", "unit_price": "100"}]
# Correct: compact JSON, no spaces
items_str = json.dumps(items, ensure_ascii=False, separators=(",", ":"))
# Result: [{"description":"Product name","quantity":"1","unit_price":"100"}]

Incorrect example (using str() directly):

# Wrong: Python's str() includes spaces, causing signature mismatch
items_str = str(items)
# Result: [{'description': 'Product name', 'quantity': '1', 'unit_price': '100'}]

For detailed signature algorithm and multi-language examples, see the Digital Signature documentation.

6.6 Response Fields

After a successful order, the server returns a checkout redirect link. Note: the actual returned field name is pay_url (snake_case), please use this field name for parsing in your code.

{
  "message": "SUCCESS",
  "pay_url": "https://gateway.next-apis.com/service-openApi/hosted/checkout/v4/...",
  "order_no": "ORDER20260710001",
  "transaction_id": "FTP202607101520240001",
  "nonce_str": "xxxxxxxx",
  "sign": "XXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXX"
}

After obtaining pay_url, redirect the user to this URL. The user can then complete the payment on the platform-hosted checkout page.
