1 Introduction

1.1 Document Overview

Card-not-present payment refers to: payment methods where the payer can transfer acceptable monetary claims to the payee through channels such as the internet, mobile internet, and wireless local area networks, without requiring a physical card to be read by acceptance devices (ATM, POS, etc.). The card-not-present payment interface is suitable for scenarios where payment is made without a physical card, such as purchasing train tickets online, online shopping, third-party platform payments, etc. Through card-not-present payment, cardholders' credit card transactions can be facilitated, reducing the number of transaction steps. At the same time, it also meets public consumption needs.

Merchants use the interface provided by the platform for integration. The merchant generates order parameters and provides them to the service provider to invoke payment. After successful payment, the merchant receives relevant payment information.

Scenario example:

Payment process:

Step (1): The user clicks submit order on the PC or mobile commerce platform.

Step (2): The order amount is displayed and card-not-present payment is selected.

Step (3): Enter the credit card number, expiration date, security code, and other security verification information, then click pay.

Step (4): The payment institution deducts the amount and displays the payment result page.

1.2 Target Audience

For reference by technical or business personnel of merchant platform service providers.

5 API Description

5.1 Create Order API

5.1.1 Business Function

After the merchant calls the Create Order API, the system generates response parameters and returns them to the merchant. The merchant determines the payment channel to invoke the payment institution for payment operations. After payment is completed, the payment institution calls back the platform, and the platform will notify the merchant interface with the results.

5.1.2 Interaction Mode

Request: Backend synchronous request mode

Return result + Notification: Backend request interaction mode + Backend notification interaction mode

5.1.3 Request Parameters

Request url: https://api.fintechpaymenthub.com/openapi/v1/pay-gui.html

Request via POST FORM submission

Field Variable Required Type Description
API type service Yes String(40) Fixed: pay.globalpayment.card.notpresent
Client ID client_id Yes String(32) Merchant client ID
Store ID store_id Yes String(50) Store ID
Merchant order no order_no Yes String(32) Order number within the merchant system, within 32 characters, can contain letters, ensure uniqueness in the merchant system
Notification URL notify_url Yes String(255) URL to receive platform notifications, must be an absolute path, within 255 characters, format like: http://wap.tenpay.com/tenpay.asp, ensure the platform can access this address via the internet
Callback URL callback_url Yes String(255) Return to the page specified by the merchant after transaction payment is completed, must be an absolute path, within 255 characters, ensure the platform can access this address via the internet. This field is only supported by Hong Kong wallets
Amount amount Yes String(15) Total order amount (must be greater than 0), in yuan
Currency currency Yes String(3) 3-digit currency code
Product description body No String(127) Product description information
Terminal IP client_ip No String(16) Machine IP where the order was generated
Random string nonce_str Yes String(32) Random string, no longer than 32 bits, different values must be passed for each submission
Signature sign Yes String(32) See Chapter 4 "Signature Rules"
Language lang No String(32) Language

5.1.4 Return Result

Payment results are notified to the interface via notification method. After payment is completed, it will redirect to the provided callback URL.

5.4 Payment Notification API

5.4.1 Business Function

After payment is completed, the system will notify relevant data to the notify_url filled in during order creation. The notification URL is the notify_url parameter submitted in the payment API. After payment is completed, the platform will send relevant payment and user information to this URL,

The merchant needs to receive and process the information.

For backend notification interaction, if the platform receives a response from the merchant that is not the pure string "success" or returns after more than 5 seconds, the platform considers the notification failed. The platform will indirectly re-initiate notifications through a certain strategy (notification frequency is 0/15/15/30/180/1800/1800/1800/1800/3600, unit: seconds) to improve the success rate of notifications as much as possible, but does not guarantee that the notification will ultimately succeed.

Since there are cases of resending backend notifications, the same notification may be sent to the merchant system multiple times. The merchant system must be able to correctly handle duplicate notifications.

The recommended approach is: when receiving a notification for processing, first check the status of the corresponding business data to determine whether the notification has already been processed. If it has not been processed, proceed with processing; if it has been processed, directly return success. Before performing status checks and processing on business data, use data locks for concurrency control to avoid data confusion caused by function re-entry.

Special note: After receiving notification parameters on the merchant backend, verify the order number order_no and order amount amount in the received notification parameters against the orders and amounts in your own business system. Only update the database order status after verification is consistent. Backend notifications are made through the notify_url in the request, returning data streams via POST method. The specific information is an XML format string (merchants should pay attention to this when processing)
