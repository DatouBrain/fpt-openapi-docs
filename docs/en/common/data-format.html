Data Format

Doc Version: V1.0

This page describes the data transmission format for the Fintech Payment OpenAPI, including request format, response format, and common parameters.

Request Format

The API supports two request submission modes:

Standard Mode (JSON POST)

The vast majority of payment channels submit data using HTTPS POST + JSON format:

  • Request Method: POST
  • Content-Type: application/json
  • Character Encoding: UTF-8
  • Data must be signed
JSON{
  "service": "pay.weixin.web.intl",
  "order_no": "00f78e479fe74a11a73be0feb9999749",
  "body": "測試支付",
  "amount": 1.00,
  "callback_url": "https://your-domain.com/result",
  "notify_url": "https://your-domain.com/notify",
  "client_ip": "127.0.0.1",
  "client_id": "1002",
  "store_id": "1001002",
  "nonce_str": "ptf3tyjv",
  "sign": "64950ED7B13489114C671EFD24C45AC3",
  "lang": "en_US"
}

Hosted Mode (FORM POST)

Global Payments Online hosted mode uses an HTML form submission, suitable for redirecting directly to the payment page in a browser:

  • Request Method: POST
  • Content-Type: application/x-www-form-urlencoded
  • Character Encoding: UTF-8
  • Data must be signed
HTML<form action="https://uat.fintechpaymenthub.com/openapi/v1/pay" method="post">
  <input type="hidden" name="service" value="pay.globalpayment.card.notpresent">
  <input type="hidden" name="client_id" value="1001">
  <input type="hidden" name="store_id" value="1001001">
  <input type="hidden" name="order_no" value="ORDER001">
  <input type="hidden" name="amount" value="100">
  <input type="hidden" name="sign" value="XXXXX">
</form>

Response Format

All endpoints return JSON format data. Determine whether the request succeeded based on the message field:

Success Response

JSON{
  "message": "SUCCESS",
  "order_no": "00f78e479fe74a11a73be0feb9999749",
  "transaction_id": "FTP202012181649370001",
  "submit_time": "20201218164937",
  "nonce_str": "o66w9pft",
  "sign": "3F6C05280C90E5BB1330524FD61C6121"
}

A message of SUCCESS indicates the API call succeeded. The response contains a sign field, which the merchant should verify to ensure data integrity.

Error Response

JSON{
  "message": "Missing Client ID",
  "errors": [
    {
      "field": "client_id",
      "message": "Missing Client ID",
      "code": "MISSING_FIELD"
    }
  ]
}

A message other than SUCCESS indicates an API call error. The errors array contains the specific error fields and error codes.

Common Request Parameters

The following parameters are common to all endpoints:

ParameterTypeRequiredDescriptionIn Signature
serviceString(40)RequiredPayment service identifier, distinguishes payment channels✅
client_idString(32)RequiredMerchant client ID✅
store_idString(50)RequiredStore ID✅
order_noString(32)RequiredMerchant internal order no, must be unique✅
nonce_strString(32)RequiredRandom string, prevents replay attacks✅
signString(64)RequiredDigital signature, see Digital Signature❌
langString(10)OptionalLanguage setting, e.g. en_US, zh_CN❌
